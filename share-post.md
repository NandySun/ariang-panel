# 我给 aria2 写了套 UI，然后被 iframe 和 CSS 轮番羞辱

如果你也用 HanaAgent，可能见过这个插件——以前叫 AriaNg，就是一个 iframe 把第三方页面嵌进来，丑、卡、操作还别扭。改了好几次都懒得推，直到上周实在忍不了，花了两天重新写了一遍。

结果两天里有大半天在跟浏览器吵架。

## 第一回合：CORS 把我锁死在门外

插件的页面跑在 Hana 的 iframe 里，要跟 aria2 的 RPC 端口（默认 6800）通信。按理说就一个 fetch 的事吧？

浏览器拦了。同源策略，跨域请求，你的 iframe 没有权限发这个包。

试了好几个方案：改 aria2 的 CORS 配置、用 JSONP、让 Hana 开白名单……全都走到半路就卡住。最后在 Hana 的插件路由层加了 `/rpc` 代理，前端所有请求发到同源的 `/rpc`，服务端转发给 aria2。

这还没完。代理路由需要认证凭据，但 iframe 里的 fetch 默认不带页面的 URL query string。返回 403，显示一个漂亮的红色错误提示。我把页面 URL 的 query 参数注入到前端的 `__AR2_URL__` 里，让请求路径带上认证信息。通了。

一个小细节卡了快两个小时。事后看就是一行代码的事，当时找不到就是找不到。

## 第二回合：每 2 秒闪一次，谁受得了

任务列表要实时刷新，标准做法是轮询。2 秒一次 aria2 的 RPC 接口，取回活动/等待/已完成的任务，重绘。

一跑起来，卡成 PPT。准确说是每 2 秒闪一次。

仔细看，问题出在我每次重绘整个列表——所有卡片重新渲染，hover 边框闪一下，展开的详情缩回去了，按钮的位置跳一下。

修了三个点：

第一，所有 hover 的 transition 动画（`border-color`、`transform`、`opacity`）全干掉。轮询场景过渡动画就是灾难。

第二，展开状态不存 DOM 里，用 JS 的一个 `_expanded` 字典追踪。每张卡片有个唯一 gid，轮询时不重置这个字典，展开的卡就保持展开。这个改了之后舒服很多，至少不会自动缩回去了。

第三，时间显示从轮询里拆出来，单独起一个 1 秒的定时器更新。轮询只管数据，时间让它自己跑。彻底消除了因为更新时间导致的"帧跳动"。

全部改完，列表终于安静地躺在那里，数据悄悄更新，页面一动不动——这才是轮询该有的样子。

## 改动过程的一些感受

原本的插件叫 AriaNg，但写着写着觉得不对——我又没用 AriaNg 的代码，自己从头搭的，凭什么还叫 AriaNg。改成"下载"，两个字，侧边栏看着舒服多了。

功能是渐进加进来的。先跑通核心面板，再补卡片展开详情（连接数、做种数、上传速度），再加右下角快捷添加按钮，最后才搞定批量操作（全部暂停/继续/清除已完成）。

交互也改了三四轮。点击卡片任意位置展开详情、点空白处收起、操作按钮和卡片点击互不干扰——这些细节看上去是"顺手做的"，其实每一条都是翻过车的结果。

还有个坑是 dev 插件会覆盖社区版。装了个社区的 AriaNg 插件，再装我自己的开发版，侧边栏开关就不工作了。排查了半天发现是 dev slot 的路由优先级高于正常插件，把社区版的路由覆盖了——不是 bug，是 Hana 的开发模式机制就这样。知道原因之后反而觉得自己蠢。

## 技术底牌

整个插件零外部依赖。前端纯 HTML/CSS/JS，服务端一个代理路由，manifest 一百多行。主题色跟随 Hana 自动切换明暗，写了完整的两套 CSS 变量。说实话写 CSS 变量比写逻辑花的时间还多。

代码放 GitHub 了，需要的自取：github.com/NandySun/ariang-panel
